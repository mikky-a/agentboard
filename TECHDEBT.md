# Техдолг и известные проблемы

Не про код доски, а про среду, в которой живут агенты. Записано, наблюдаем.

## Cursor и opencode: границы интеграции (2026-07-20)

- «Продолжить» после паузы умеет только claude: карточки codex/cursor/opencode
  без sessionId исчезают с доски после смерти tmux-сессии. У всех троих resume
  есть в CLI (`codex resume`, `cursor-agent --resume <chatId>`, `opencode -s <id>`),
  id мы уже храним в `card["rollout"]` — осталось дописать resume_card.
- Cursor CLI игнорирует ответ `{"permission":"allow"}` из beforeShellExecution-хука
  (in-process runner). Поэтому tee-имя разрешено через allowlist `Shell(tee)`
  в `~/.cursor/cli-config.json`. Если починят — вернуть точечное разрешение в хук.
- У Cursor нет события «жду разрешения»: waiting ставится перед каждым shell/MCP
  и снимается после. Долгая авто-разрешённая команда какое-то время выглядит
  как «ждёт» (панель-эвристика это гасит).
- opencode с дефолтными правами почти не спрашивает разрешений — plugin ловит
  permission.asked, но в тесте событие не стрельнуло ни разу. Проверить на
  строгом konfigе (permission.bash = "ask").
- Первый запуск Cursor в новой папке — диалог Workspace Trust в карточке,
  одно нажатие «a». Trust запоминается per-папка.

## «API Error: Stream idle timeout - no chunks received» (2026-07-19)

Сессия внезапно останавливается после tool-вызова: следующий запрос к API
открывается, но ~3 минуты не приходит ни одного чанка, встроенный watchdog
Claude Code (в 2.1.215 дефолт ~180 с) убивает поток, ход завершается ошибкой,
агент ждёт ручного «продолжи». За 19.07 случилось дважды в одной длинной
сессии (534 сообщения в контексте) — похоже либо на зависшее соединение
(VPN/прокси), либо на долгий первый байт при пересборке кэша большого контекста.

Известная проблема Claude Code, полного фикса нет:
- https://github.com/anthropics/claude-code/issues/46987 (и #47698, #53730)
- корень частично в SDK: watchdog не видит ping-события —
  https://github.com/anthropics/anthropic-sdk-typescript/issues/998

Если достанет — смягчение в `~/.claude/settings.json` (переменные проверены
по бинарнику 2.1.215):

```json
"env": {
  "CLAUDE_STREAM_IDLE_TIMEOUT_MS": "300000",
  "CLAUDE_CODE_MAX_RETRIES": "15"
}
```

300000 — потолок, зашитый в код; больше нельзя. Есть ещё
`CLAUDE_ENABLE_STREAM_WATCHDOG=0`, но с ним мёртвое соединение будет висеть
дольше, не трогать без нужды. Статус: пока не применяли, смотрим частоту.

## Баннер «✘ Auto-update failed» в tmux-сессиях доски (2026-07-19)

Сам автоапдейт работает: версии качаются в день релиза в
`~/.local/share/claude/versions/`, симлинк `~/.local/bin/claude` обновляется,
ручной `claude update` из tmux доски проходит. PATH доска прописывает сама
(AGENT_PATH в agentboard.py) — tmux ни при чём.

Баннер — от разовой фоновой проверки, упавшей по сети: в
`~/.claude/.last-update-result.json` лежит `install_failed` с
`version_to: null`, т.е. запрос к downloads.claude.ai не дошёл вообще
(та же нестабильная сеть, что и у stream timeout выше). Дальше косметический
баг Claude Code: результат «up to date» при следующей удачной проверке баннер
не сбрасывает — красная надпись висит до перезапуска сессии, хотя всё свежее.

Со стороны доски не чинится. Если надоест: ставить `DISABLE_AUTOUPDATER=1`
агентам с доски и обновлять централизованно (кроном `claude update`) — тогда
баннеров не будет совсем. Второе следствие: долгоживущая сессия работает на
той версии, с которой стартовала; новая подхватывается только при перезапуске.
